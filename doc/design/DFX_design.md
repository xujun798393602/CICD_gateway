## 1. 稳定性分析（Stability）

### 1.1 故障模式分析

| 故障点 | 故障类型 | 影响范围 | 严重程度 |
|--------|----------|----------|----------|
| 单个扫描脚本崩溃 | 进程异常退出 | 该扫描器结果缺失，其余正常 | 中 |
| 扫描脚本超时（>10分钟） | 资源耗尽 | 阻塞流水线，占用CI Runner | 高 |
| GitLab API 不可达 | 网络/服务故障 | 无法获取仓库信息、无法写MR评论 | 高 |
| 报告生成服务异常 | 内部服务故障 | 无法生成HTML报告 | 中 |
| 数据库连接失败 | 基础设施故障 | 扫描结果无法持久化 | 严重 |
| 邮件/通知服务不可用 | 外部依赖故障 | 通知无法发送 | 低 |

### 1.2 容错机制

**扫描脚本隔离执行**
- 每个扫描器在独立子进程中执行，任何一个扫描器的失败不影响其他扫描器
- 退出码映射：`0` = 成功，`1` = 发现问题，`2` = 执行错误，`124` = 超时
- 任一扫描器失败后，将其状态标记为 `failed` 或 `skipped`，继续执行后续扫描器

**超时处理**
- 单个扫描器默认超时：600 秒（10 分钟），可配置
- 超时触发后，发送 `SIGTERM`，等待 5 秒后若未退出则发送 `SIGKILL`
- CI 流水线侧有 15 分钟总超时保护

**报告生成降级**
- HTML 报告生成失败时，自动降级为 JSON 格式报告
- JSON 报告包含与 HTML 报告相同的数据结构，仅缺少可视化渲染
- 降级过程对用户透明，响应中增加 `degraded: true` 标识

### 1.3 验收标准

- 单个扫描器崩溃时，其余扫描器 100% 正常完成
- 超时扫描器在超时阈值 + 5 秒内被强制终止
- HTML 报告生成失败时，JSON 降级报告在 10 秒内可用

## 2. 可用性分析（Availability）

### 2.1 服务健康检查

健康检查接口 `GET /api/v1/health` 覆盖以下组件：

| 组件 | 检查方式 | 健康判定 | 降级行为 |
|------|----------|----------|----------|
| 数据库 | 执行 `SELECT 1` 查询 | 响应 < 500ms | 只读模式 |
| 缓存（Redis） | `PING` 命令 | 响应 < 200ms | 绕过缓存直接查库 |
| 存储（报告文件） | 写入/读取/删除测试文件 | 操作成功 | JSON降级 |
| GitLab API | 请求 `/api/v4/version` | 响应 < 2s | GitLab功能暂停 |

### 2.2 故障传播控制

- 扫描执行是核心路径，不依赖报告生成和通知发送
- 报告生成失败不阻塞扫描结果的记录和门禁判定
- 通知发送失败不阻塞任何其他功能

### 2.3 验收标准

- 健康检查接口响应时间 < 100ms
- 单组件故障时，系统整体可用（降级模式下）
- 故障组件恢复后，系统在 30 秒内自动恢复正常模式

## 3. 可扩展性分析（Scalability）

### 3.1 并发扫描能力

| 资源 | 单任务消耗 | 5 并发需求 | 10 并发需求（预留） |
|------|-----------|-----------|-------------------|
| CPU | 1 核 | 5 核 | 10 核 |
| 内存 | 512MB | 2.5GB | 5GB |
| 数据库连接 | 2 | 10 | 20 |

**并发控制机制**
- 同一项目同一分支不允许重复扫描（返回 40901 错误码）
- 全局并发扫描数上限通过配置控制，默认 10
- 达到上限时，新任务进入排队状态

### 3.2 插件化架构

**扫描器插件规范**

| 规范项 | 要求 |
|--------|------|
| 入口脚本 | 可执行文件，支持命令行调用 |
| 输入约定 | 通过环境变量接收参数 |
| 输出约定 | 通过标准输出输出 JSON 格式结果 |
| 退出码 | `0` = 成功无问题，`1` = 成功有问题，`2` = 执行错误 |
| 超时处理 | 脚本需处理 `SIGTERM` 信号 |

**新扫描器接入流程**
1. 开发扫描脚本，遵循上述规范
2. 将脚本放置到指定目录（如 `scanners/`）
3. 调用注册接口 `POST /api/v1/scanners/register` 完成注册
4. 在项目配置中启用该扫描器
5. 下次扫描时自动执行

### 3.3 验收标准

- 5 个项目并发扫描，单项目扫描时间增加 < 10%
- 新扫描器从注册到首次执行的端到端时间 < 5 分钟
- 配置变更后无需重启服务，下次扫描即生效

## 4. 性能分析（Performance）

### 4.1 性能指标

| 指标 | 目标值 | 预算分解 |
|------|--------|----------|
| 单文件扫描 | ≤ 30 秒 | 文件读取 < 1s + 规则匹配 < 25s + 结果输出 < 4s |
| 10 万行代码库完整扫描 | ≤ 10 分钟 | 文件遍历 < 30s + 并行扫描 < 8min + 结果汇总 < 30s |
| 报告生成 | ≤ 1 分钟 | 数据查询 < 5s + 模板渲染 < 50s + 文件写入 < 5s |
| API 响应时间（P99） | ≤ 500ms | 数据库查询 < 200ms + 业务逻辑 < 200ms + 序列化 < 100ms |

### 4.2 优化策略

**扫描执行优化**
- 增量扫描：仅扫描变更文件（通过 `git diff` 获取变更文件列表）
- 并行扫描：多个扫描器并行执行（独立进程），使用 `asyncio.create_subprocess_exec`
- 文件过滤：默认排除 `node_modules/`、`.git/`、`vendor/`、`build/` 等目录
- 规则预编译：正则表达式在启动时预编译，运行时复用

**报告生成优化**
- 异步生成：报告生成接口立即返回 `report_id`，后台异步处理（FastAPI BackgroundTasks）
- 模板缓存：Jinja2 模板预加载到内存
- 分页加载：大量问题列表分页渲染

**API 性能优化**
- 数据库索引：`scan_id`、`project_id`、`created_at` 建立复合索引
- 连接池：SQLAlchemy async engine 连接池大小 = 并发数 * 2
- 结果缓存：Redis 缓存扫描状态查询结果，TTL 5 秒
- 异步IO：全链路使用 async/await，避免阻塞事件循环

### 4.3 验收标准

- 单文件扫描时间 ≤ 30 秒
- 10 万行代码库完整扫描 ≤ 10 分钟
- 报告生成 ≤ 1 分钟
- API P99 响应时间 ≤ 500ms

## 5. 安全性分析（Security）

### 5.1 凭证管理

| 凭证类型 | 存储方式 | 轮换策略 |
|----------|----------|----------|
| GitLab API Token | GitLab CI/CD Variable | 每 90 天轮换 |
| Git 仓库 SSH Key | GitLab CI/CD Variable | 每 180 天轮换 |
| 邮件服务密码 | 服务端环境变量 | 每 90 天轮换 |
| 数据库密码 | 服务端环境变量 | 每 90 天轮换 |

### 5.2 敏感信息脱敏

| 敏感信息类型 | 脱敏规则 |
|-------------|----------|
| 密码（password=xxx） | 保留键名，值替换为 `***` |
| API 密钥 | 保留前 4 位和后 4 位，中间替换 |
| 私钥 | 整行替换为 `[REDACTED: PRIVATE KEY]` |
| 数据库连接串 | 密码部分替换为 `***` |

排除规则（不标记为敏感词）：
- 环境变量引用格式：`${VARIABLE_NAME}`、`$VARIABLE_NAME`
- 测试代码目录中的内容（可通过配置白名单排除）

### 5.3 脚本执行隔离

| 隔离维度 | 实现方式 |
|----------|----------|
| 进程隔离 | 每个扫描器在独立子进程中执行 |
| 文件系统隔离 | 每次扫描在临时目录中克隆代码，扫描完成后清理 |
| 资源限制 | CPU 限制 1 核、内存限制 1GB、磁盘限制 5GB |
| 超时保护 | 单个扫描器最长执行 10 分钟，超时强制终止 |

### 5.4 报告访问控制

| 访问场景 | 控制方式 |
|----------|----------|
| Web 报告页面 | GitLab OAuth2 认证，仅项目成员可访问 |
| API 报告查询 | Bearer Token 认证，Token 与项目权限绑定 |
| 报告下载链接 | 带签名的临时 URL，有效期 24 小时 |

### 5.5 验收标准

- 凭证 100% 通过 CI/CD Variables 管理，代码仓库零硬编码
- 报告中敏感信息脱敏率 100%
- 扫描器进程无法访问沙箱外的文件系统和网络

## 6. 可维护性分析（Maintainability）

### 6.1 结构化日志

所有组件输出 JSON 格式的结构化日志：

```json
{
  "timestamp": "2024-01-15T10:30:00.123Z",
  "level": "INFO",
  "module": "scan_orchestrator",
  "scan_id": "scan_20240115_abc123",
  "request_id": "req_abc123def456",
  "message": "Scan task created",
  "duration_ms": 45
}
```

日志级别使用规范：
- **DEBUG**：扫描器详细执行过程
- **INFO**：扫描任务创建/完成、报告生成、通知发送
- **WARN**：扫描器超时（但已处理）、降级触发、重试
- **ERROR**：扫描器失败、报告生成失败、通知发送失败
- **FATAL**：数据库连接失败、服务启动失败

### 6.2 扫描结果可追溯

每次扫描分配全局唯一的 `scan_id`，格式：`scan_{YYYYMMDD}_{8位随机字符串}`

通过 `scan_id` 可追溯一次扫描的完整生命周期：触发时间、各扫描器执行状态、发现的问题、生成的报告、发送的通知。

### 6.3 监控指标暴露

通过 `GET /api/v1/metrics` 接口暴露 Prometheus 格式指标：

| 指标名称 | 类型 | 说明 |
|----------|------|------|
| `quality_gate_scans_total` | Counter | 扫描总次数 |
| `quality_gate_scan_duration_seconds` | Histogram | 扫描耗时分布 |
| `quality_gate_scan_errors_total` | Counter | 扫描错误次数 |
| `quality_gate_reports_generated_total` | Counter | 报告生成总次数 |
| `quality_gate_api_request_duration_seconds` | Histogram | API 请求耗时 |

### 6.4 验收标准

- 所有日志输出为 JSON 格式，包含 timestamp、level、module、request_id、message 五个必填字段
- 通过 scan_id 可在 1 分钟内追溯一次扫描的完整执行链路
- 监控指标覆盖扫描、报告、API 三个维度

## 7. 兼容性分析（Compatibility）

### 7.1 GitLab 版本兼容

| 特性 | GitLab 14.x | GitLab 15.x | 兼容策略 |
|------|-------------|-------------|----------|
| CI/CD Variables | 支持 | 支持 | 使用标准 API |
| MR API | `/api/v4/...` | 同左 | 使用 v4 API |
| OAuth2 | 支持 | 支持 | 使用标准 OAuth2 流程 |

**兼容性实现原则**：
- 仅使用 GitLab v4 REST API，不使用特定版本的私有 API
- 启动时通过 `GET /api/v4/version` 检测 GitLab 版本，自动选择适配器

### 7.2 编程语言支持

| 语言 | 扫描器 | 代码规范工具 | 依赖 |
|------|--------|-------------|------|
| Java | Checkstyle, SpotBugs | Checkstyle | JDK 11+ |
| Python | Pylint, Bandit | Pylint | Python 3.8+ |
| JavaScript | ESLint | ESLint | Node.js 16+ |
| TypeScript | ESLint + @typescript-eslint | ESLint | Node.js 16+ |

### 7.3 浏览器兼容

| 浏览器 | 最低版本 | 支持范围 |
|--------|----------|----------|
| Chrome | 90+ | 完全支持 |
| Firefox | 88+ | 完全支持 |
| Edge | 90+ | 完全支持 |

**报告页面兼容策略**：
- 使用 ES2020 语法，不使用实验性特性
- CSS 使用 Flexbox/Grid 布局
- 不依赖 JavaScript 框架，使用原生 HTML/CSS/JS

### 7.4 验收标准

- GitLab 14.x 和 15.x 环境中，CI/CD 集成全流程测试通过率 100%
- Java/Python/JavaScript/TypeScript 四种语言的扫描功能正常
- Chrome/Firefox/Edge 三种浏览器报告页面渲染一致

## 风险识别与缓解措施汇总

| 风险 ID | 风险描述 | 影响程度 | 缓解措施 |
|---------|----------|----------|----------|
| R-001 | GitLab API 版本升级导致接口不兼容 | 高 | API 版本适配层 + GitLab 升级前回归测试 |
| R-002 | 扫描器脚本存在安全漏洞被利用 | 高 | 沙箱隔离 + 资源限制 + 定期安全审计 |
| R-003 | 大型代码库扫描超时 | 中 | 增量扫描 + 并行执行 + 超时配置可调 |
| R-004 | 敏感信息脱敏遗漏导致信息泄露 | 高 | 双层脱敏（扫描器 + 服务端）+ 定期脱敏规则审计 |
| R-005 | 并发扫描导致数据库连接池耗尽 | 中 | 连接池监控 + 动态扩容 + 排队机制 |
